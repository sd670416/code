# 低代码 SaaS 工作流平台技术方案

## 1. 建设目标

建设一个面向多租户的低代码平台，支持租户下创建多个应用。每个应用具备独立的菜单、角色、权限、表单模型、业务数据和工作流配置。平台通过拖拽方式完成表单设计、菜单配置、权限划分和流程编排，最终形成可运行的业务系统。

本方案已结合旧项目 `F:\project\smart`、`F:\project\gongcheng` 和 `F:\project\vite-vue-bpmn-process-dev` 重新设计。新版采用 `F:\project\smart` 作为基础框架参考，主平台使用 JDK 21 / Spring Boot 4.1，工作流采用外置 Camunda 7 服务承接二次开发能力。完整重设计说明见 `docs/redesigned-solution.md`，Camunda 二次开发细节见 `docs/camunda-extension-design.md`，最终设计汇总见 `docs/final-design-summary.md`，开发规范见 `docs/development-standards.md`。

核心能力：

- 应用管理：租户内创建应用，应用之间数据、菜单、权限隔离。
- 表单设计：拖拽生成表单、列表、详情页，支持字段、校验、布局、数据源配置。
- 工作流配置：集成 Camunda，通过 BPMN 配置审批流、任务节点、条件分支、候选人规则。
- 权限体系：支持菜单权限、操作权限、数据权限，账号可绑定多个角色。
- 数据隔离：平台级租户隔离，应用级业务隔离，业务数据按 `tenant_id` + `app_id` 过滤。
- SaaS 运维：支持租户、应用、用户、角色、审计、日志、版本发布等平台化治理。

## 2. 技术栈

后端：

- JDK 21
- Spring Boot 4.1
- Spring Security
- MyBatis-Plus 或 Spring Data JDBC
- MySQL 8
- Redis
- 外置 Camunda 7 工作流服务

前端：

- Vue 3
- Vite
- TypeScript
- Pinia
- Vue Router
- Naive UI
- 表单设计器：基于 JSON Schema / 自定义组件协议封装
- 流程设计器：基于 BPMN.js 集成 Camunda BPMN XML

前端部署原则：

- 用户只进入一套主平台前端 `code/web`。
- 平台管理、应用管理、表单运行、流程设计、任务中心都在同一个登录态、同一套路由和同一套菜单权限中。
- BPMN 设计器和流程配置页面直接集成在 `code/web`，不再创建 Camunda 侧前端目录。

推荐工作流集成方式：

- 采用外置 Camunda 7 工作流服务。主平台 `api` 不直接集成 Camunda starter，而是通过 HTTP 客户端调用 `camunda`。
- 平台保存流程定义元数据、版本、发布状态和业务绑定关系。
- 流程运行推进以 Camunda 为准，平台侧保存业务实例、任务快照、审批记录、节点按钮、表单权限和权限辅助数据。
- 平台需要建设 Camunda 二次开发扩展层，统一封装审批动作，包括办理、退回至上一节点、退回至发起人、任意退回、终止、加签、减签、转办、委托、解决、拿回。

Camunda 版本第一版建议采用 Camunda 7.19+。旧系统的退回、任意退回、拿回、多实例处理依赖 Camunda 7 的流程实例修改能力，第一版不建议切换到 Camunda 8。

## 3. 总体架构

```text
web/vue3 + naive ui
  |
  | REST API / WebSocket
  v
api/springboot4.1
  |
  |-- IAM 权限模块
  |-- 租户与应用模块
  |-- 菜单与页面模块
  |-- 表单设计模块
  |-- 动态数据模块
  |-- 工作流编排模块
  |-- 审计日志模块
  |-- 工作流平台编排模块
  |
  |-- MySQL 8
  |-- Redis
  |
  v
camunda/springboot2 + camunda7
  |
  |-- Camunda Engine
  |-- Camunda MySQL
```

主平台后端采用模块化单体优先。工作流引擎单独拆出，是为了兼容 Camunda 7 与 Spring Boot 4.1/JDK 21 的版本边界，同时保留旧系统已验证的审批动作能力。

## 4. 后端分层设计

```text
controller
  处理 HTTP 入参、鉴权上下文、返回 DTO

application/service
  编排业务流程，处理事务、权限校验、跨模块调用

domain
  领域模型、状态流转、规则判断

infrastructure
  数据库访问、Redis、Camunda Client、文件存储、消息通知
```

建议模块：

- `system`：租户、用户、部门、角色、字典、参数。
- `iam`：登录认证、菜单权限、操作权限、数据权限。
- `app`：应用、应用版本、应用资源、应用发布。
- `form`：表单模型、字段模型、页面布局、组件配置。
- `data`：动态数据表、记录、字段值、数据导入导出。
- `workflow`：流程定义元数据、流程绑定、流程实例、任务快照、审批记录、调用外置工作流服务。
- `integration`：`camunda` 客户端、回调验签、同步修复。
- `audit`：登录日志、操作日志、数据变更日志。

## 5. 多租户和应用隔离方案

采用共享数据库、共享表、字段隔离方式：

- 所有平台业务表必须包含 `tenant_id`。
- 所有应用内资源表必须包含 `app_id`。
- 动态业务数据必须同时包含 `tenant_id`、`app_id`、`form_id`。
- 后端从登录态中解析 `tenant_id`，普通接口禁止客户端自行传入租户上下文。
- 应用级接口需要校验当前用户是否拥有该 `app_id` 的访问权限。
- 数据访问层统一追加租户和应用过滤条件，避免业务代码遗漏。

隔离层级：

```text
tenant
  -> user / dept / tenant role
  -> app
       -> app role
       -> menu
       -> form
       -> workflow
       -> business data
```

## 6. 认证与权限设计

认证：

- 用户使用账号密码登录。
- 登录成功后签发 JWT access token 和 refresh token。
- Redis 保存会话、刷新令牌、验证码、黑名单和在线状态。

授权：

- 用户可绑定多个角色。
- 角色分为平台角色和应用角色。
- 平台角色管理租户级能力，例如租户管理员、平台运维。
- 应用角色管理应用内菜单、按钮、数据和流程权限。

权限类型：

- 菜单权限：控制侧边栏、路由、页面入口。
- 操作权限：控制新增、编辑、删除、导入、导出、提交、审批等按钮或 API。
- 数据权限：控制用户可见、可编辑的数据范围。

数据权限规则建议支持：

- 本人数据。
- 本部门数据。
- 本部门及下级部门数据。
- 指定部门数据。
- 指定用户数据。
- 表单字段条件，例如 `region = 当前用户.region`。
- 自定义 SQL 片段不直接开放给前端，只允许平台管理员通过安全模板配置。

## 7. 低代码表单设计

表单设计分三层：

- 表单元数据：表单编码、名称、版本、状态、所属应用。
- 字段元数据：字段编码、字段类型、控件类型、校验规则、默认值、是否列表展示。
- 页面 Schema：布局、分组、栅格、组件属性、联动规则。

字段类型：

- 文本、长文本、数字、金额、日期、日期时间。
- 单选、多选、下拉、开关。
- 附件、图片。
- 用户、部门、角色选择器。
- 关联表单、子表、明细表。

业务数据存储建议分阶段：

第一阶段采用通用记录表 + JSON 数据：

- 开发快，适合表单频繁变化。
- MySQL JSON 字段保存表单值。
- 对高频查询字段额外写入索引表。

第二阶段支持物理表生成：

- 为稳定表单生成独立物理表。
- 查询性能更好。
- 需要处理字段变更、版本兼容和 DDL 发布。

建议先实现第一阶段，并预留第二阶段能力。

## 8. 工作流设计

流程设计：

- 前端基于 `vite-vue-bpmn-process-dev` 改造成平台内 BPMN 设计器。
- 平台扩展节点属性面板，例如候选人、候选角色、表单权限、按钮权限、超时规则。
- 保存 BPMN XML、流程配置 JSON 和版本信息。
- 发布流程时由 `api` 解析扩展配置并落库，再调用 `camunda` 部署到 Camunda，并记录 Camunda process definition id/key/version。
- 节点扩展配置包括候选人、表单权限、按钮权限、意见必填、附件要求、退回范围、加签减签策略。

流程绑定：

- 一个表单可以绑定一个或多个流程。
- 可按场景绑定，例如请假单普通审批、请假单补卡审批。
- 表单支持无流程、可选流程、必须流程三种模式。
- 无流程表单提交后只保存业务记录，不创建流程实例。
- 必须流程表单支持保存草稿，发起时才创建流程实例。
- 可选流程表单由业务场景或提交入口决定是否发起流程。

表单提交模式：

| 模式 | 保存草稿 | 提交后行为 | 适用场景 |
| --- | --- | --- | --- |
| `NONE` | 可选 | 保存为普通业务数据，不进入 Camunda | 台账、档案、基础数据 |
| `OPTIONAL` | 可选 | 可直接提交，也可按场景发起审批 | 部分场景需要审批的业务 |
| `REQUIRED` | 必须支持 | 先保存业务数据，再发起 Camunda 流程 | 请假、合同、费用、采购 |

任务处理：

- 待办列表从平台任务快照表查询，任务状态与 Camunda 同步。
- 任务打开时根据节点配置加载表单字段权限。
- 审批动作至少包括办理 `approve`、退回至上一节点 `reject`、退回至发起人 `rejectFirst`、任意退回 `rejectCustom`、终止 `stop`、加签 `addAssignee`、减签 `removeAssignee`、转办 `transfer`、委托 `delegate`、解决 `resolve`、拿回 `getBack`。
- 审批动作统一由后端 `FlowActionService` 执行，前端只提交动作编码、意见、目标用户、目标节点和表单数据。

流程变量：

- 表单字段可映射为流程变量。
- 发起人、部门、角色、金额、区域等常用条件必须标准化。
- 条件分支不直接读取任意 JSON，发布时生成稳定变量映射。

## 9. 前端模块设计

路由模块：

- 登录页。
- 租户管理。
- 平台菜单管理。
- 平台角色权限。
- 应用工作台。
- 应用设置。
- 应用菜单设计。
- 应用角色权限。
- 表单设计器。
- 流程设计器。
- 权限配置。
- 动态列表页。
- 动态详情页。
- 待办、已办、我发起的流程。

前端运行时：

- 登录后拉取用户信息、租户信息、应用列表、权限码。
- 登录后先加载平台级菜单，进入应用后再加载应用级菜单。
- 进入应用后拉取应用菜单和路由 Schema。
- 动态页面根据后端返回的页面 Schema 渲染。
- 所有按钮通过操作权限码控制显示和禁用。
- 表单运行页根据 `workflow_mode` 渲染按钮：无流程显示保存、提交；有流程显示保存草稿、发起流程；审批待办显示节点按钮。
- 基础组件统一使用 Naive UI，二次封装平台级表格、表单、弹窗、抽屉、权限按钮、用户选择器、部门选择器等组件。
- 主题能力基于 Naive UI theme overrides 实现，第一期提供“深海蓝”和“赛博青”两套科技感后台主题。
- 主题偏好属于用户个人偏好，登录后从用户偏好接口读取，切换后保存到 `sys_user_preference`，前端本地存储只做临时缓存。

设计器：

- 左侧组件面板。
- 中间画布。
- 右侧属性面板。
- 顶部保存、预览、发布、版本管理。

## 10. Redis 使用设计

Redis 存储：

- 登录会话。
- refresh token。
- 验证码。
- 权限缓存。
- 应用菜单缓存。
- 数据字典缓存。
- 分布式锁，例如流程发布、表单发布。
- Camunda 任务同步临时状态。

缓存失效：

- 用户角色变化后清理用户权限缓存。
- 菜单、操作权限变化后清理应用权限缓存。
- 字典变化后清理字典缓存。
- 表单发布后清理页面 Schema 缓存。

## 11. 安全设计

- 所有接口必须认证，登录、验证码、公开资源除外。
- 后端进行接口权限校验，前端隐藏按钮仅作为体验优化。
- 租户、应用、角色、用户边界必须在服务层校验。
- 数据权限必须在查询层统一生效。
- 操作日志记录关键配置变更、流程审批、权限调整。
- 密码使用 BCrypt 或 Argon2 哈希。
- 文件上传限制类型、大小，并隔离租户目录。
- 防止低代码表达式注入，不允许前端提交任意脚本在后端执行。

## 12. 开发阶段规划

第一阶段：平台基础

- 后端工程初始化。
- 前端工程初始化。
- 用户、租户、应用、角色、菜单、权限。
- 登录认证和动态菜单。

第二阶段：表单低代码

- 表单设计器。
- 表单 Schema 保存、预览、发布。
- 通用数据新增、编辑、删除、查询。
- 列表页和详情页动态渲染。

第三阶段：工作流

- BPMN 设计器集成。
- 流程定义保存、部署、绑定表单。
- 发起流程、待办、审批、审批记录。
- 节点表单权限和按钮权限。
- 外置 `camunda` 服务。
- 迁移旧系统 `approve`、`reject`、`rejectFirst`、`stop`。

第四阶段：平台增强

- 数据权限规则。
- 应用发布和版本管理。
- 审计日志、导入导出、报表。
- 物理表生成和高级查询优化。

## 13. 关键技术风险

- 主平台不直接集成 Camunda 7，避免 Spring Boot 4.1/JDK 21 兼容性风险。
- Camunda 7 外置服务需要验证与 MySQL 8、部署环境、回调机制的稳定性。
- 通用 JSON 数据模型查询灵活但性能有限，需要字段索引表补强。
- 数据权限容易散落在业务代码中，必须抽象统一查询拦截。
- 表单版本和流程版本需要强绑定，否则历史流程打开旧表单会出错。
- 多应用菜单和权限不能只做前端过滤，后端 API 必须强校验。
