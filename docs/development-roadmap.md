# smart-code 分阶段开发计划

## 1. 开发原则

本项目按“先平台骨架，再表单，再流程，最后平台化增强”的顺序推进。每个阶段必须能独立验收，验收通过后再进入下一阶段，避免一开始同时铺开低代码、权限、流程、设计器导致范围失控。

阶段性成果以可运行、可演示、可回归为准：

- 后端接口能启动、能访问 Swagger/OpenAPI 或接口文档。
- 前端页面能进入主流程，不只停留在静态页面。
- 数据库有明确初始化脚本和测试数据。
- 每阶段保留最小测试用例，后续阶段不能破坏前面能力。
- Camunda 复杂能力按白名单逐步开放，先稳定普通流程，再开放退回、拿回、多实例、网关等高风险能力。

## 2. 总体阶段

| 阶段 | 名称 | 核心目标 | 验收方式 |
| --- | --- | --- | --- |
| 0 | 项目初始化 | 建立前后端和工作流工程骨架 | 三个工程可独立启动 |
| 1 | 基础平台 | 完成登录、租户、用户、组织、角色、菜单权限 | 管理员可配置组织和菜单 |
| 2 | 应用与表单 | 完成低代码应用、页面、表单设计、表单版本 | 可创建应用并提交一条业务数据 |
| 3 | 工作流服务骨架 | 外置 Camunda 7 服务可部署、启动、查询任务 | 可完成最简单审批流程 |
| 4 | 流程设计器集成 | 集成 BPMN 设计器和节点扩展属性 | 可在前端设计并发布流程 |
| 5 | 审批操作第一期 | 完成办理、退回上一节点、终止、拿回 | 典型串行流程闭环 |
| 6 | 任务中心与历史 | 完成待办、已办、我的流程、审批历史、任务快照 | 业务端可完整查看流程轨迹 |
| 7 | 表单权限与候选人规则 | 完成节点字段权限、候选人规则、按钮权限 | 不同节点显示不同字段和按钮 |
| 8 | 审批操作第二期 | 完成转办、委托、解决、加签、减签、退回发起人 | 支持常见 OA 审批动作 |
| 9 | 流程高级能力 | 多实例、网关、任意退回、消息同步、异常修复 | 通过专项流程测试矩阵 |
| 10 | 平台增强 | 数据权限、报表、审计、发布管理、运维监控 | 可作为 SaaS 平台试运行 |

## 3. 阶段 0：项目初始化

### 3.1 开发内容

- 创建主平台后端 `api`：
  - JDK 21。
  - Spring Boot 3。
  - MyBatis-Plus 或 MyBatis。
  - MySQL 8。
  - Redis。
  - Sa-Token 或 Spring Security。
- 创建前端 `web`：
  - Vue 3。
  - Vite。
  - TypeScript。
  - Naive UI。
  - Pinia。
  - Vue Router。
- 创建外置工作流后端 `camunda/backend`：
  - Spring Boot 2.7。
  - Camunda 7.19+。
  - 独立数据库或独立 schema。
  - 对主平台暴露 HTTP API。
- 创建流程设计器组件目录 `camunda/web`：
  - 放置 BPMN 设计器和流程配置组件源码。
  - 不作为独立用户后台。
  - 后续集成到主平台 `web`。
- 统一代码规范：
  - 后端统一异常结构。
  - 后端统一返回结构。
  - 前端统一请求封装。
  - 前端统一路由、布局、菜单渲染。

### 3.2 验收标准

- `api` 能启动并连接 MySQL、Redis。
- `web` 能启动，展示 Naive UI 主布局，并作为唯一前端入口。
- `camunda/backend` 能启动，Camunda 表自动初始化或脚本初始化成功。
- `camunda/web` 目录存在，明确为流程设计器组件源码目录。
- 三个工程都能读取本地配置和开发环境配置。
- 形成第一版 `README.md` 启动说明。

### 3.3 阶段风险

- 主平台 Spring Boot 3 与 Camunda 7 不兼容，所以 Camunda 必须外置。
- 不要在第一阶段就迁移旧审批逻辑，先把工程边界搭稳。

## 4. 阶段 1：基础平台

### 4.1 开发内容

- 租户管理：
  - 租户创建、启用、停用。
  - 租户编码唯一。
  - 所有核心表预留 `tenant_id`。
- 用户与组织：
  - 用户 CRUD。
  - 部门树。
  - 岗位。
  - 用户部门关系。
- 权限体系：
  - 角色。
  - 平台菜单。
  - 应用菜单。
  - 按钮权限。
  - API 权限。
  - 登录用户上下文。
- 前端基础布局：
  - 登录页。
  - 主框架。
  - 左侧菜单。
  - 顶部用户信息。
  - 标签页或面包屑。

### 4.2 验收标准

- 管理员可以登录。
- 管理员可以维护租户、用户、部门、角色、平台菜单、应用菜单。
- 不同平台角色登录后平台菜单不同。
- 不同应用角色进入应用后应用菜单不同。
- 后端接口能根据登录用户识别 `tenant_id`。
- 所有基础接口通过最小集成测试。

### 4.3 阶段风险

- 权限模型不要过早复杂化，第一期只实现菜单权限、按钮权限、API 权限。
- 数据权限放到后续阶段，先保证基础 RBAC 清晰。

## 5. 阶段 2：应用与表单

### 5.1 开发内容

- 应用管理：
  - 应用创建。
  - 应用菜单。
  - 应用页面。
  - 应用发布状态。
- 表单设计：
  - 表单基础信息。
  - 表单流程模式：无流程、可选流程、必须流程。
  - 表单 JSON schema。
  - 表单版本。
  - 表单发布。
- 动态数据：
  - 表单记录主表。
  - 表单记录 JSON 数据。
  - 创建、编辑、查看、删除。
  - 基础列表查询。
- 前端低代码表单：
  - 先实现运行态表单渲染。
  - 设计器可先做简化版，后续增强。
  - 使用 Naive UI 组件作为渲染基础。

### 5.2 验收标准

- 可以创建一个应用。
- 可以在应用下创建一个表单。
- 可以发布一个表单版本。
- 可以创建一个无流程表单。
- 可以通过无流程表单录入一条业务数据。
- 可以在列表中查看和编辑业务数据。
- 无流程表单提交后不创建 `flow_instance`。

### 5.3 阶段风险

- 表单数据第一期建议用 JSON 存储，不急于生成物理业务表。
- 表单版本必须从第一期就做，否则后面流程绑定会出现历史表单不可追溯问题。
- 阶段 2 只验收无流程表单的保存和提交，有流程表单只完成字段预留。

## 6. 阶段 3：工作流服务骨架

### 6.1 开发内容

- 搭建 `camunda` 独立服务。
- 封装 Camunda 基础能力：
  - 部署流程。
  - 启动流程。
  - 查询待办。
  - 完成任务。
  - 查询流程状态。
- 主平台接入工作流服务：
  - `smart-workflow` 模块只调用工作流服务 API。
  - 主平台保存业务流程实例映射。
  - 主平台保存任务快照。
- 有流程表单运行态：
  - 保存草稿。
  - 发起流程。
  - 草稿不创建 Camunda 实例。
  - 发起后创建 Camunda 实例。
- 初始化最小 BPMN：
  - 开始事件。
  - 发起人任务。
  - 审批人任务。
  - 结束事件。

### 6.2 验收标准

- 主平台可以调用工作流服务部署 BPMN。
- 主平台可以把有流程表单保存为草稿。
- 主平台可以基于草稿或新表单发起一条流程。
- 审批人能查到待办。
- 审批人点击办理后流程结束。
- 主平台能看到流程实例状态和审批日志。

### 6.3 阶段风险

- 这一阶段只做 `approve`，不要同时做退回、拿回、转办。
- Camunda 原生表不直接暴露给前端，前端只访问主平台 API。
- 必须确认无流程表单仍可独立提交，不被工作流逻辑影响。

## 7. 阶段 4：流程设计器集成

### 7.1 开发内容

- 参考 `F:\project\vite-vue-bpmn-process-dev` 集成 BPMN 设计器。
- 改造成新平台组件：
  - 使用 Naive UI 弹窗、表格、表单。
  - 保留 `camunda:smartProperty` 扩展思路。
  - 支持流程保存、预览、发布。
  - 组件源码放在 `camunda/web`，运行入口集成到主平台 `web`。
- 节点扩展属性：
  - 按钮配置 `buttonsSetting`。
  - 表单字段权限 `formEditable`。
  - 候选人规则。
  - 监听器配置。
- 流程发布：
  - 校验 BPMN。
  - 生成流程定义版本。
  - 部署到 Camunda。

### 7.2 验收标准

- 可以在前端新建流程模型。
- 可以设置用户任务节点的审批按钮。
- 可以设置节点表单字段只读、可编辑、隐藏。
- 可以发布流程并生成平台流程定义版本。
- 发布后的流程可以被发起。

### 7.3 阶段风险

- 设计器不要直接调用 Camunda，统一提交到主平台，由主平台转发给 `camunda/backend`。
- 不做第二套流程管理前端，流程页面统一挂在主平台菜单下。
- BPMN 扩展属性要有版本兼容策略，避免后续字段改名导致旧流程无法解析。

## 8. 阶段 5：审批操作第一期

### 8.1 开发内容

第一期审批动作只做低风险闭环：

- 办理 `approve`。
- 退回上一节点 `reject`。
- 退回发起人 `rejectFirst`。
- 终止 `stop`。
- 拿回 `getBack`。

后端设计：

- 每种审批动作独立 `ActionHandler`。
- 增加任务操作幂等：
  - `request_id`。
  - Redis task lock。
  - 操作日志唯一约束。
- 增加任务权限校验：
  - 是否当前办理人。
  - 是否当前租户。
  - 当前节点是否允许该按钮。
  - 当前业务数据是否可操作。
- 增加任务快照同步：
  - Camunda 待办变化后同步到平台表。

### 8.2 验收标准

- 串行三节点流程可完整办理。
- 第二节点可以退回上一节点。
- 审批人可以在后续节点未处理前拿回。
- 发起人或管理员可以终止流程。
- 重复点击审批按钮不会产生重复历史。
- 非当前审批人不能办理任务。

### 8.3 阶段风险

- 拿回只开放“当前任务仍由下一处理人持有且未完成”的场景。
- 退回上一节点只先支持普通串行和简单排他网关。
- 并行网关、多实例、子流程先不开放退回和拿回。

## 9. 阶段 6：任务中心与历史

### 9.1 开发内容

- 任务中心：
  - 我的待办。
  - 我的已办。
  - 我的发起。
  - 抄送我的。
  - 流程详情。
- 审批历史：
  - 操作人。
  - 操作类型。
  - 审批意见。
  - 操作时间。
  - 来源节点。
  - 目标节点。
- 流程轨迹：
  - 当前节点高亮。
  - 已完成节点。
  - 被退回节点。
  - 终止状态。

### 9.2 验收标准

- 用户能在待办中处理任务。
- 已办能看到自己处理过的任务。
- 发起人能看到自己发起的流程状态。
- 流程详情能展示审批历史和当前节点。
- 终止、退回、拿回都有清晰历史记录。

### 9.3 阶段风险

- 审批历史不能只依赖 Camunda 历史表，必须有平台自己的不可变业务日志。
- 任务快照和审批日志职责要分开，避免旧代码中历史表同时承担显示和状态判断的问题。

## 10. 阶段 7：表单权限与候选人规则

### 10.1 开发内容

- 节点表单权限：
  - 字段可见。
  - 字段隐藏。
  - 字段只读。
  - 字段必填。
- 候选人规则：
  - 指定用户。
  - 指定部门。
  - 指定角色。
  - 指定岗位。
  - 发起人上级。
  - 表单字段关联用户。
- 按钮权限：
  - 节点允许哪些按钮。
  - 按钮自定义名称。
  - 按钮是否必须填写意见。

### 10.2 验收标准

- 不同审批节点看到不同字段状态。
- 指定候选人规则能正确生成待办。
- 未配置的审批按钮不会显示。
- 必填意见的按钮不填写意见时不能提交。

### 10.3 阶段风险

- 候选人规则要落库保存解析结果，不能只在 Camunda 变量里临时存在。
- 表单权限要基于表单版本，不要直接绑定最新表单 schema。

## 11. 阶段 8：审批操作第二期

### 11.1 开发内容

- 转办 `transfer`。
- 委托 `delegate`。
- 解决 `resolve`。
- 加签 `addAssignee`。
- 减签 `removeAssignee`。
- 撤销或作废 `cancel`。
- 完善动作处理器：
  - 参数校验。
  - 权限校验。
  - Camunda 操作。
  - 平台快照同步。
  - 审批日志。
  - 业务回调。

### 11.2 验收标准

- 当前审批人可以转办给其他用户。
- 委托人可以委托，被委托人处理后可解决。
- 多实例节点可以加签、减签。
- 所有动作都有审批历史。
- 所有动作都有接口测试。

### 11.3 阶段风险

- 加签、减签只对多实例节点开放。
- 委托、解决要明确 Camunda 原生语义，不要和转办混用。

## 12. 阶段 9：流程高级能力

### 12.1 开发内容

- 支持多实例：
  - 会签。
  - 或签。
  - 通过比例。
- 支持网关：
  - 排他网关。
  - 并行网关。
  - 包容网关。
- 支持任意退回：
  - 可退节点计算。
  - 退回路径校验。
  - 网关安全校验。
- 支持异常修复：
  - 流程实例同步。
  - 任务快照重建。
  - 失败回调重试。
  - 管理员干预日志。

### 12.2 验收标准

- 多实例会签流程通过专项测试。
- 并行网关流程通过专项测试。
- 任意退回不会破坏执行树。
- 任务快照与 Camunda 运行态一致。
- 异常同步任务可以修复不一致数据。

### 12.3 阶段风险

- 任意退回是最高风险能力，必须最后做。
- 每开放一种流程结构，都要加入测试矩阵。

## 13. 阶段 10：平台增强

### 13.1 开发内容

- 数据权限：
  - 本人数据。
  - 本部门数据。
  - 本部门及下级数据。
  - 自定义数据范围。
- 审计日志：
  - 登录日志。
  - 操作日志。
  - 流程干预日志。
- 报表统计：
  - 流程数量。
  - 待办数量。
  - 超时任务。
  - 平均审批耗时。
- 运维监控：
  - 服务健康检查。
  - Camunda 同步状态。
  - Redis 状态。
  - 失败任务告警。

### 13.2 验收标准

- 租户之间数据完全隔离。
- 管理员可以查看关键操作日志。
- 可以查看流程统计。
- 可以发现和处理工作流同步异常。

## 14. 推荐开发顺序

实际开发建议严格按下面顺序推进：

1. 阶段 0：项目初始化。
2. 阶段 1：基础平台。
3. 阶段 2：应用与表单。
4. 阶段 3：工作流服务骨架。
5. 阶段 4：流程设计器集成。
6. 阶段 5：审批操作第一期。
7. 阶段 6：任务中心与历史。
8. 阶段 7：表单权限与候选人规则。
9. 阶段 8：审批操作第二期。
10. 阶段 9：流程高级能力。
11. 阶段 10：平台增强。

不建议跳过阶段 3 直接做流程设计器，也不建议跳过阶段 5 直接做复杂退回。工作流的稳定性来自最小闭环不断扩展，而不是一次性把所有审批按钮搬完。

## 15. 第一轮里程碑

第一轮建议做到阶段 5，形成可演示 MVP：

- 管理员登录。
- 创建应用。
- 创建表单。
- 创建流程。
- 流程绑定表单。
- 发起流程。
- 审批人办理。
- 退回上一节点。
- 终止流程。
- 拿回流程。
- 查看审批历史。

第一轮暂不做：

- 任意退回。
- 并行网关退回。
- 包容网关退回。
- 子流程。
- 复杂数据权限。
- 报表统计。
- 租户计费。

## 16. 每阶段交付物

每个阶段完成后至少交付：

- 代码变更。
- 数据库迁移脚本。
- 初始化测试数据。
- 接口说明。
- 前端演示页面。
- 阶段验收记录。
- 已知问题清单。

阶段验收记录建议放在：

```text
code/docs/stage-checks/
```

每个阶段一个文件，例如：

```text
stage-00-init.md
stage-01-platform.md
stage-02-form.md
stage-03-camunda.md
```

## 17. 当前建议立即开始的任务

下一步建议从阶段 0 开始：

1. 创建 `api` 后端工程骨架。
2. 创建 `web` 前端工程骨架，接入 Naive UI。
3. 创建 `camunda/backend` 外置 Camunda 7 工程骨架。
4. 创建 `camunda/web` 流程设计器组件目录。
5. 编写基础数据库初始化脚本。
6. 编写本地启动说明。

阶段 0 完成后再进入阶段 1，不提前开发审批动作。
